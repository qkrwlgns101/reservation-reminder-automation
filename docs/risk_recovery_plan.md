# 장애·실수·데이터 꼬임 대응 설계

## 목적

이 문서는 예약·리마인더 자동화 패키지를 실제 고객에게 납품할 때 발생할 수 있는 최악의 상황을 미리 정의하고, 감지·복구·예방 방법을 정리한 운영 설계서입니다.

핵심 원칙은 아래와 같습니다.

1. 자동화보다 데이터 보존이 우선이다.
2. 실제 발송 전에는 직원 확인 단계를 둔다.
3. 오류는 조용히 무시하지 않고 기록한다.
4. 문제가 생기면 마지막 정상 상태로 되돌릴 수 있어야 한다.
5. 민감정보는 처음부터 받지 않는 구조로 줄인다.

## 최악 상황 목록

| 상황 | 위험 | 예방 장치 | 복구 방법 |
|---|---|---|---|
| 네트워크 끊김 | 폼 제출 실패, 시트 동기화 지연 | 직원용 수기 접수 양식 준비 | 네트워크 복구 후 수기 접수분 입력 |
| Google 서비스 장애 | 폼/시트 접근 불가 | 오프라인 접수 템플릿 준비 | 장애 종료 후 백업 기준으로 재입력 |
| 직원이 수식 삭제 | 발송 문구/리포트 오류 | 보호 범위, 백업, 사전 점검 메뉴 | 백업 시트 또는 원본 템플릿에서 복구 |
| 직원이 상태값 오타 입력 | 집계 오류 | 드롭다운, 사전 점검 | 잘못된 상태값 Validation_Errors에서 수정 |
| 중복 예약 입력 | 발송 중복, 리포트 과대 집계 | appointment_id 중복 점검 | 중복 행 병합 또는 취소 처리 |
| 날짜/시간 잘못 입력 | 잘못된 발송 대기 생성 | 날짜 필드 검증 | 예약일/시간 수정 후 대기함 재생성 |
| 수신동의 누락 | 불필요한 연락 위험 | 동의 FALSE는 보류 처리 | 동의 확인 후 대기 상태로 변경 |
| 담당자가 발송완료 체크 누락 | 중복 발송 위험 | 발송 로그, 처리자 기록 | 실제 발송 내역 확인 후 상태 보정 |
| 동시 편집 | 데이터 충돌, 중복 생성 | LockService 잠금 | Audit_Log 확인 후 중복 제거 |
| Apps Script 실행 실패 | 대기함/리포트 미갱신 | 오류 기록, 수동 실행 메뉴 | 오류 메시지 확인 후 재실행 |
| 문자/API 연동 실패 | 실제 메시지 미발송 | 1차 버전은 수동 발송 | 발송 대기함에서 재처리 |
| 고객정보 유출 | 신뢰·법적 리스크 | 최소 수집, 권한 제한 | 접근권한 회수, 로그 확인, 고객 고지 |

## 네트워크 장애 운영 모드

### 증상

- Google Form이 열리지 않는다.
- 응답 제출 후 저장 여부가 불확실하다.
- Google Sheets가 오프라인 상태로 보인다.
- Apps Script 메뉴 실행이 실패한다.

### 직원 행동 지침

1. Google Form 제출을 반복하지 않는다.
2. 예약 정보를 임시 수기 접수표에 기록한다.
3. 고객에게는 “예약 요청을 접수했고 확인 후 확정 안내드리겠다”고 안내한다.
4. 네트워크 복구 후 수기 접수표를 `Appointments`에 직접 입력한다.
5. `예약 자동화 > 사전 점검 실행`을 실행한다.
6. 이상이 없으면 `발송 대기함 갱신`을 실행한다.

### 오프라인 수기 접수 항목

| 항목 | 필수 |
|---|---|
| 접수 시각 | 예 |
| 고객명 | 예 |
| 연락처 | 예 |
| 예약 희망일 | 예 |
| 예약 희망시간 | 예 |
| 예약 항목 | 예 |
| 수신 동의 여부 | 예 |
| 메모 | 선택 |

## 사람 실수 대응 원칙

### 직원이 직접 수정해도 되는 영역

- `Settings`의 사업장 정보
- `Appointments`의 예약 데이터
- `Message_Templates`의 문구 본문
- `Send_Queue`의 발송상태, 확인자, 비고

### 직원이 수정하면 안 되는 영역

- 대시보드 수식
- 주간 리포트 수식/자동 계산 영역
- Apps Script 코드
- 숨겨진 보조 시트
- 예약 ID, 고객 ID, queue_id 규칙

## 데이터 꼬임 감지 기준

사전 점검에서 아래 항목을 확인합니다.

- 필수 시트 존재 여부
- 필수 헤더 존재 여부
- 예약 ID 누락
- 예약 ID 중복
- 고객명 누락
- 연락처 누락
- 예약일 누락 또는 잘못된 형식
- 상태값 오류
- 수신동의 값 오류
- 발송 대기함 중복
- 템플릿 누락

오류는 `Validation_Errors` 시트에 기록합니다.

## 백업 정책

### 수동 백업

고객에게 아래 상황에서는 반드시 백업을 만들도록 안내합니다.

- 처음 납품 직후
- 문구 템플릿을 크게 바꾸기 전
- 기존 예약 데이터를 대량 업로드하기 전
- Apps Script를 수정하기 전
- 월말 리포트 마감 전

### 자동화 백업

Apps Script 메뉴의 `백업 스냅샷 생성` 기능은 주요 시트를 복사해 날짜가 붙은 백업 시트를 만듭니다.

백업 대상:

- `Settings`
- `Appointments`
- `Message_Templates`
- `Send_Queue`
- `Weekly_Report`

### 월별 아카이브 파일

Apps Script 메뉴의 `월별 아카이브 파일 생성` 기능은 현재 스프레드시트 전체를 Google Drive의 별도 파일로 복사합니다.

권장 실행 시점:

- 매월 말 또는 매월 1일
- 월간 리포트 확정 후
- 오래된 예약 데이터를 정리하기 전
- 담당자 변경 또는 외부 유지보수 종료 전

운영 기준:

- 무료판은 월 예약 300건 이하, 누적 예약 3,000행 이하 사용을 권장합니다.
- 누적 데이터가 많아지면 월별 아카이브 파일을 만든 뒤 오래된 행을 정리합니다.
- 아카이브 파일도 개인정보 파일이므로 공유 권한과 보관 기간을 반드시 관리합니다.

자세한 기준은 [`무료 사용 한도와 백업 정책`](usage_limits_and_backup_policy.md)을 참고합니다.

## 권한 관리

### 권장 권한

| 사용자 | 권한 |
|---|---|
| 대표/원장 | 편집자 |
| 실장/관리자 | 편집자 |
| 일반 직원 | 제한된 편집자 또는 댓글/보기 |
| 외부 작업자 | 구축 기간에만 편집자 |

### 납품 후 권장 조치

- 외부 작업자 권한 회수
- 고객 소유 Google 계정으로 파일 이전
- Apps Script 소유자 확인
- 공유 링크가 `링크가 있는 모든 사용자`로 열려 있지 않은지 확인

## 장애 발생 시 고객 안내 문구

```text
현재 Google 서비스 또는 네트워크 상태로 인해 예약 자동화 반영이 지연될 수 있습니다.
예약 정보는 임시 접수표에 기록한 뒤, 연결이 복구되는 즉시 시스템에 반영하겠습니다.
예약 확정 여부는 담당자가 별도로 안내드리겠습니다.
```

## 납품 전 필수 테스트

아래 테스트를 통과한 뒤 고객에게 납품합니다.

- 정상 예약 3건 입력
- 수신동의 FALSE 예약 1건 입력
- 예약일이 오늘인 예약 1건 입력
- 예약일이 내일인 예약 1건 입력
- 상태가 미방문인 예약 1건 입력
- 중복 appointment_id 입력 후 오류 감지 확인
- 잘못된 상태값 입력 후 오류 감지 확인
- 발송 대기함 갱신 2회 실행 후 중복 생성 방지 확인
- 백업 스냅샷 생성 확인
- 주간 리포트 갱신 확인
